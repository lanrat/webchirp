## 2026-07-23

- Loading a `.img` file could not reliably detect the radio: CHIRP's `directory.get_radio_by_image()` only searches drivers already present in `DRV_TO_RADIO`, and the browser runtime imports driver modules lazily (the static catalog means none are imported at boot). An image whose metadata trailer names a not-yet-imported driver failed with `Unsupported model`, and a metadata-less image could only byte-match against whatever happened to be imported. Fix: `handleLoadImage` now parses the metadata trailer first (`read_image_metadata_base64`, driver-free), resolves the driver module from the radio catalog (`findCatalogRadioForImageMetadata`: exact `rclass` class-name match, falling back to vendor/model), and imports that module before running detection.
- CHIRP image metadata's `rclass` can name a class that is not directory-registered: `uv5r.BaofengUV5R` is an unregistered base class (the registered UV-5R entry is `BaofengUV5RGeneric`), yet `export_image_base64` happily builds images from it and stamps `rclass: BaofengUV5R`. Detection then resolves such images via the metadata vendor/model match, not the class name — so catalog matching must keep a vendor/model fallback and must not assume `rclass` exists in the catalog. Also note `directory.MODEL_COMPAT` remaps renamed vendor/model pairs (e.g. Retevis RT-5R → RT5R) and must be applied when reading metadata.
- Channel paste silently lost values in columns the grid marks read-only: `normalizeValue()` returned `previous` for `editable: false` columns, so pasted rows (built from blank defaults) kept the default instead of the pasted value. On the Baofeng UV-5R (`has_tuning_step = False` in `uv5r.py`) every pasted channel's TStep reset to `2.50` (first enum option) regardless of the copied `5.00`. Same mechanism affects `Comment` on radios with `has_comment=False` and any `has_ctone`/`has_dtcs`/`has_offset`-gated column on radios that disable them. Fix: programmatic row builders (paste, repeater/GMRS imports) pass `allowReadOnly` so kind/options validation still applies but the editable gate doesn't; interactive grid editors keep the gate.
- Numeric enum columns (`TStep` "5.00", `rToneFreq` "88.5", `DtcsCode` "023") rejected spreadsheet-normalized values like "5", "12.5", "23" because enum matching was exact-string only; values now fall back to numeric-equality matching against the option list before reverting to the default.
- The `eaeba93` upstream merge took master's `package.json` wholesale, silently dropping `scripts/test-clipboard.mjs` and `scripts/test-ui-channel-cut.mjs` from `npm run test:channels` — the branch's clipboard tests existed but were not being run. Merge resolutions that "take theirs" on `package.json` need a check that branch-added test entries survive.
- A sibling of the PR #8 pinned-`base.sha` bug: after pushing new commits to PR #7's head branch (`lanrat/feature/channel-copy-paste`), the fork branch and `GET /compare/master...lanrat:feature/channel-copy-paste` both showed the new tip immediately, but the PR's `headRefOid` stayed pinned at the previous push for many minutes — GitHub's async PR-sync job simply never fired, so the PR page showed no new commits. This is persistent, not a one-off: every subsequent push to the branch exhibited the same stuck head, each needing the workaround again. The same workaround fixes it: retarget the PR's base to any other branch and back (`gh pr edit N --base <other>` then `--base master`), which re-records both base and head refs. Diagnostic order that isolates it: `git ls-remote` (did the push land?) → compare API (does GitHub see the commits?) → PR `headRefOid` (did the PR sync?).

## 2026-07-22

- GitHub Pages serves everything with a fixed, non-configurable `Cache-Control: max-age=600` + ETag (via Fastly); custom headers cannot be set. Compared to the CloudFront setup in `../webchirp-hosting` (assets `max-age=3600, must-revalidate`, `index.html` `max-age=0, must-revalidate`, invalidation on deploy), this creates a ≤10-minute post-deploy window where a stale cached `index.html` can reference old content-hashed asset names that no longer exist in the new Pages artifact, 404ing JS/CSS until the cache expires. Self-healing and low-impact at current deploy frequency; if it ever matters, the fix is to make `build:dist` retain the previous generation of hashed assets in the artifact rather than trying to control headers. Performance is otherwise fine: revalidation is a cheap ETag 304, and the heavy payloads (Pyodide, CHIRP sources) come from jsDelivr with long cache lifetimes, not from Pages.
- The repo's GitHub Pages was previously a legacy Jekyll build of raw `master`: it served the Jekyll-rendered README at `/` and the unbuilt app at `/web/`, with `https_enforced` off even though Web Serial/WebUSB require a secure context. Switched to workflow builds (`.github/workflows/pages.yml` deploying `dist/`) with HTTPS enforced at `https://webchirp.jasiek.me/`.
- GitHub pins a PR's diff base (`base.sha`) to the merge-base computed when the PR was created and never advances it — not when the base branch moves, and not even when the head branch merges in the newer base. PR #8 kept showing RELEASE_NOTES.md +3 and web/about.html +1 (the favicon-subpath changes already merged to master via PR #11) even though the head's file blobs were byte-identical to the current merge-base, and pushing a fresh commit to the head updated `head.sha` but left `base.sha` pinned. The fix that works: retarget the PR's base branch to any other branch and back (`gh pr edit N --base <other>` then `--base master`), which re-records `base.sha` as the current merge-base and drops the phantom files from the diff. `GET /repos/{owner}/{repo}/compare/master...<head>` always computes a fresh merge-base, so it is the reliable way to see a PR's true current diff.

## 2026-06-23

- First-load latency for the radio make/model dropdowns was dominated by `list_registered_radios()` importing *every* CHIRP driver (191 modules) in Pyodide. Each not-yet-cached import trips the `ChirpCdnFinder` hook, which does one blocking jsDelivr fetch per module file (plus its dependencies) via `_await_js`, so the dropdowns could not appear until ~191 serialized fetch→write→import→exec cycles (on top of Pyodide cold start) completed. The catalog only needs vendor/model/module/class/baud/isLive per radio, none of which requires running the driver in the browser.
- Fix: precompute the catalog at build time from the local CHIRP submodule (`scripts/build-catalog.mjs` reusing the `TestRadioHarness` filesystem source) into `web/radio-catalog.json`, and have `handleListRadios` fetch that static file first, falling back to live in-browser enumeration only if it is missing/unusable. Pyodide now boots lazily (on first radio selection / CSV / serial action) instead of blocking the initial list render.
- Generating the catalog from the submodule revision can drift from the browser runtime's pinned `CHIRP_REVISION`, and a drifted catalog can expose driver classes the runtime doesn't have (or hide ones it does). The runtime now accepts the static catalog only when its `chirpRevision` matches the pin exactly (falling back to live enumeration otherwise), the pin lives in one shared constant (`DEFAULT_CHIRP_REVISION` in `web/js/python-sources.mjs`), and `build:catalog` refuses to emit a catalog whose submodule revision disagrees with that pin.
- Selecting a radio fired `getRadioMetadata` and `getRadioSettings` concurrently, and both import the selected driver module. The lazy `ChirpCdnFinder` import hook suspends the interpreter mid-import (JSPI `run_sync`) while fetching module source, so the second import re-executes the same module body before the first finishes; CHIRP's `directory.register` then raises `Duplicate radio driver id`. The race pre-dates the catalog work (reproduced on upstream master) and was reproduced deterministically both in Node with `--experimental-wasm-stack-switching` and in headless Chrome. Concurrent RPCs also clobbered the shared `_sel_*` interpreter globals.
- Fix: every Pyodide-backed RPC now runs through a FIFO call queue (`web/js/call-queue.mjs`), so no two calls can interleave inside the interpreter; only `getRuntimeInfo` bypasses the queue because error reporting needs it while a call is stuck. On the UI side, metadata/settings loads carry a monotonically increasing token and stale responses are discarded, so an older radio's response can never overwrite state for a newer selection; search keystrokes debounce the load and skip it when the effective selection did not change.

## 2026-03-14

- `buttons.github.io/buttons.js` is not a viable GitHub star widget under cross-origin isolation. Even if the script is self-hosted, the widget path still depends on cross-origin embed resources, which conflicts with `COEP`/`COOP` on the static `codeplug.org` deployment.
- The bug-report CTA was previously hidden until debug logs matched `error|traceback|exception`, which made manual reports impossible for usability bugs or flows that fail without those exact markers. The report action should stay visible and prefill a generic issue even with no selected radio or captured runtime error.
- Some CHIRP drivers do not instantiate cleanly with `None` when enumerating metadata or settings from a non-downloaded state. Generic runtime helpers should fall back to `radio_cls("")` to preserve best-effort schema loading before a live download.
- Clone-mode settings introspection is not safe to assume on blank/default state. `get_radio_settings()` can fail for drivers like `baofeng_uv17Pro.UV17Pro` unless the runtime has first been seeded with a real cached image or download result.
- An all-radios initial-load sweep shows the browser prep path is broader than the `UV17Pro` failure: several drivers currently throw during `get_radio_settings()` on blank/default state, including missing parsed memory objects, missing `_` translation bindings, and serialization of `None` setting values.
- Root cause: browser startup was treating CHIRP radio settings as static metadata. In practice many drivers only materialize settings from loaded backing state: a cached clone image, a parsed file image, or a live serial session. Initial UI load should treat settings as unavailable until that backing state exists, instead of eagerly calling `get_settings()`.
- Radio-settings validation was replaying every serialized value onto a fresh CHIRP settings tree before upload, including immutable fields that the UI correctly disabled. Drivers can reject those no-op assignments with `This value is not mutable`, so bridge-side apply logic must skip non-mutable CHIRP values entirely.
- Vendored CHIRP's `RadioSettingValueFloat.get_max()` currently returns `None` because the method body has no `return`. Runtime code that serializes float setting bounds must tolerate that upstream defect and fall back to the instance `_max` field.
- `webchirp-hw-*` temp directories were created by older real-radio CLI test infrastructure in commit `1f820a5`, but that code was removed in `b456a75`. The remaining `.gitignore` entry was stale residue; current tracked code no longer creates repo-root hardware temp directories.
